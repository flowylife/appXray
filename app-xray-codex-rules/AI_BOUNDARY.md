# AI Boundary Specification

## Purpose

This document defines what AI should and should not do in App X-Ray.

The main design rule:

> AI suggests. The user confirms. Logic preserves.

## AI-Owned Responsibilities

AI is useful when natural language must be interpreted.

### 1. Input Understanding

AI can read:

- rough app ideas
- PRDs
- notes
- meeting summaries
- feature lists
- domain-specific descriptions

AI should output structured suggestions.

### 2. App Type Classification

AI can classify the app into one or more app types:

- internal tool
- dashboard
- asset management
- inspection app
- scheduler
- CRM
- inventory app
- marketplace
- CMS
- booking app
- personal productivity app

The result should include:

- primary type
- secondary types
- confidence
- short explanation

### 3. Requirement Extraction

AI can extract:

- screen requirements
- feature requirements
- data requirements
- permission requirements
- flow requirements
- non-functional requirements
- ambiguous statements

### 4. Structure Suggestion

AI can suggest:

- screens
- screen groups
- features
- data objects
- data fields
- data relations
- user roles
- permissions
- flows
- flow steps
- issues

### 5. Missing Parts Detection

AI can detect:

- missing requirements
- ambiguous terms
- contradictions
- missing permissions
- missing data fields
- missing status values
- missing exception handling
- oversized scope
- unsupported assumptions

### 6. Plain-Language Explanation

AI should explain technical findings in non-developer language.

Example:

Bad:
> RBAC policy is incomplete.

Good:
> 누가 설비 정보를 수정할 수 있는지 아직 정해지지 않았습니다.

### 7. Build Prompt Generation

AI can generate prompts for:

- Codex
- Cursor
- Lovable
- Replit
- Bolt

Prompt generation must use confirmed structured project data as input.

## Logic-Owned Responsibilities

These must be deterministic and should not rely on AI.

### 1. Project Storage

- create project
- update project metadata
- save source documents
- save analysis results
- save user edits
- delete project

### 2. Object Lifecycle

- suggested
- accepted
- edited
- rejected
- deferred

### 3. Graph Rendering

- node layout
- edge rendering
- zoom/pan
- selection state
- filters
- detail panels

### 4. Validation

Examples:

- screen name cannot be empty
- data object name must be unique within a project
- flow must have at least two steps
- relation cannot reference missing data objects
- accepted export cannot include rejected objects

### 5. Export

Generate from confirmed data:

- Markdown
- Mermaid
- JSON
- GitHub issue markdown
- prompt source package

### 6. Version Diff

Logic should compute structural differences.

AI may explain impact after logic computes the diff.

## AI Output Contract

AI should return strict JSON for analysis tasks.

The app should validate AI output before saving it.

Invalid AI output must not crash the app.

## Suggested Analysis JSON Shape

```json
{
  "summary": {
    "appName": "string",
    "appTypes": ["string"],
    "confidence": 0.0,
    "plainLanguageSummary": "string"
  },
  "requirements": [],
  "screens": [],
  "features": [],
  "dataObjects": [],
  "dataRelations": [],
  "roles": [],
  "permissions": [],
  "flows": [],
  "issues": [],
  "buildPlan": []
}
```

## AI Safety Rules for Product Consistency

1. Never invent hidden confirmed data.
2. Mark inferred items as inferred.
3. Preserve source references when possible.
4. Use stable IDs generated by logic, not by AI, whenever possible.
5. If AI returns IDs, treat them as temporary IDs.
6. Never override user-edited names.
7. Never remove objects without user confirmation.
8. Never include rejected objects in generated prompts.

## Confidence Handling

Use confidence internally.

Recommended confidence bands:

- `0.80-1.00`: likely
- `0.50-0.79`: review
- `0.00-0.49`: weak suggestion

Visible UI labels:

- likely: "추천"
- review: "확인 필요"
- weak suggestion: "약한 추정"

## Prompt Generation Boundary

Prompt generator input should be assembled by logic:

- selected screens
- selected data objects
- selected flows
- selected issues
- selected target AI tool
- selected build step
- excluded scope

AI then converts this structured package into a clear instruction.

## Anti-Patterns

Do not implement these patterns:

```text
Raw PRD → AI → final Mermaid export
Raw PRD → AI → final Codex prompt with invented features
User edits node → next AI run deletes or renames it
AI response text becomes canonical project state
AI decides billing, access, or deletion
```

Preferred pattern:

```text
Raw PRD
→ AI structured suggestions
→ validation
→ user review
→ accepted/edited canonical structure
→ deterministic export
→ optional AI wording
```
